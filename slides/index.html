01
§ 1 · the gap

Most LLM usage today is fundamentally stateless.

We send a prompt, the model returns an answer, and the moment that response is delivered, every signal we could have learned from is gone. The same question tomorrow produces an answer with the same gaps, because nothing in the system has changed.

02
§ 1 · the shift

The real shift isn't to better prompts or bigger models.
It's to better systems.

Systems where LLMs operate inside closed loops, produce persistent artifacts, evaluate their own outputs, and get better over time without anyone retraining anything.

The future of AI systems is probably less about bigger models, and more about systems that improve themselves over time. Familiar engineering principles (state, evaluation, iteration, observability) applied to a new kind of system.

03
The model is increasingly becoming
the least interesting component.

04
§ 2 · defining the loop

A self-improving system is a closed loop.

act
→
evaluate
→
persist
→
adapt
↻
01 · ACT

Do the work the user asked for. Return a result. The model usually lives here.

02 · EVALUATE

Turn the unstructured result into a structured signal. Code, regex, or a schema-constrained model call.

03 · PERSIST

Write the signal somewhere durable. JSON, database, log. The request ends; the signal does not.

04 · ADAPT

Build the next prompt from the new state. The behaviour changes because the state changed, not the model.

The model does one part of one step. The system does everything else.

05
§ 3 · the dichotomy

Retrieval ≠ Behavioral adaptation.

retrievalInject context per query.

Per-request lookup. Same model, same behavior style. Documents in, answer out.

  • Context is fetched, then thrown away
  • Next request starts fresh
  • The system doesn't change
  • Useful, necessary, not the same thing

behavioral adaptationState mutates the system.

Outcomes update persistent state. The next prompt is constructed differently because of what happened.

  • Feedback compounds into structure
  • Future behavior depends on past behavior
  • The system changes
  • This is the thing that improves
06
Retrieval alone is not self-improvement.
Behavioral adaptation is.

§ 4 · canonical example

karpathy/autoresearch: the loop in four files.

program.md
orchestration / skill
Plain-English instructions for the agent. The "research org code."
→ human edits this
train.py
behavioral adaptation
The full GPT model + optimizer + loop. Architecture and hyperparams: all fair game.
→ agent edits this
5-min budget + val_bpb
evaluator
Fixed wall-clock budget per experiment. One scalar metric, lower is better. Deterministic.
→ keep or discard
results.tsv
memory / trace
Every experiment, every metric. The agent reads its own history.
→ next iteration uses this

Human edits program.md. Agent edits train.py. Neither edits the model.

§ 5 · the demo

Same idea. Smaller scope. ~170 lines of Python.

Local model via LM Studio. JSON on disk for memory. No frameworks. No vector DB. The architecture is the lesson.

planner.txt
orchestration
System prompt that consumes {memory_block}. Asks the model to answer the user, respecting whatever memory contains.
evaluator.txt
evaluator
System prompt with strict JSON schema. Reads user feedback, emits structured preferences.
memory.json
memory
The persistent state. Likes, dislikes, constraints, notes. Mutated by every feedback call.
demo.py
runtime / trace
~170 lines. Four commands: ask / feedback / show / reset. Prints every step of the loop.

[ runs entirely on this laptop · open-source model (phi-4-mini) · open-source code · no cloud calls ]

§ 5 · what just happened

Same prompt. Same model. Same temperature.

Ask #1 · empty memory
(no preferences known yet)
"Consider trying an outdoor adventure like hiking at your local state park, or exploring nature trails to enjoy fresh air and beautiful scenery while staying active."
Ask #2 · after feedback
dislikes: outdoor activities
constraints: tight budget
"Consider visiting your local library's free events section or exploring an online virtual museum tour to enjoy culture and learning without spending money."

The prompt template re-rendered because memory.json mutated. That's it. That's the loop.

11
§ 5 · be precise

This demo shows adaptation. Not improvement.

The behavior changed. That's adaptation. But there's no goal-aligned metric, no convergence, nothing that lets me say the system is getting better.

Improvement = adaptation + a goal-aligned evaluator + iteration that converges. Autoresearch supplies the missing piece. This demo is the substrate.

credibility moveSaying this out loud, before the objection lands, is worth more than the demo itself.

§ 6 · the dark side

When the loop hurts you.

Failure mode What goes wrong
feedback poisoningOne bad signal contaminates all future behavior. No quarantine, no rollback.
memory driftSlow accumulation of contradictions. Union-merge never forgets, never resolves.
stale memoryPreferences captured a year ago drive today's behavior. Recency unweighted.
hallucination persistenceThe evaluator extracted a confident lie. Now it's "fact" in memory forever.
echo chambersThe system narrows. Surprise disappears. Personalization becomes a cage.
evaluator failureThe LLM judging the LLM is wrong, confidently and silently. Turtles all the way down.
bad adaptation loopsThe system optimises for the signal you measured, not the goal you wanted. Goodhart, at runtime.

§ 7 · where this lands

Most production AI systems improve
at the orchestration layer.

  • Persistent skills accumulate. Reusable workflows compound.
  • Eval-driven prompt iteration replaces ad-hoc prompt engineering.
  • Execution traces become the dataset for the next round of iteration.
  • The interesting question is no longer “which model?” It's “do we have the right loop?”
